# 도매몰 운영 A/B 테스트 체크리스트 (운영팀용 · 코드 불필요)

> "A안 vs B안"을 **운영자가 직접 나눠서 돌려보고, 어드민 숫자로 더 나은 쪽을 고르는** 실전 가이드.
> 개발/통계 지식 필요 없음. 어드민 대시보드 + 메시지/알림/안내만으로 진행.

---

## 한 장 요약 (이 순서대로)

1. **무엇을 비교할지 1가지** 정한다 (예: 환영 메시지 A vs B)
2. **두 그룹으로 나눈다** (예: 가입순번 홀수=A / 짝수=B, 또는 주차 번갈아)
3. **한쪽만 바꾼다** (나머지는 똑같이 — 안 그러면 뭐가 효과인지 모름)
4. **기간을 정한다** (최소 2주, 짧으면 운 / 길면 정확)
5. **볼 숫자 1개**를 정한다 (예: 첫 발주율 — 어드민에서 확인)
6. 끝나고 **A vs B 숫자 비교** → 더 나은 쪽 채택, 큰 차이 없으면 유지
7. **기록표에 남긴다** (맨 아래 템플릿) — 다음에 또 헷갈리지 않게

> 💡 차이가 **확실히 크게(예: 1.3배 이상)** 나야 채택. 5%, 10% 차이는 "운"일 수 있으니 한 번 더 돌려보기.

---

## 무엇으로 그룹을 나눌까 (운영자가 수동으로)

같은 조건의 두 집단을 만드는 게 핵심. 편한 방법 아무거나:

- **가입 순번 홀짝** — 홀수 회원=A안, 짝수=B안 (가장 공평·간단)
- **주차 번갈아** — 이번 주 신규=A, 다음 주 신규=B (계절 효과 주의)
- **지역/등급** — 단, 지역·등급은 원래 성향이 다를 수 있어 비교 왜곡 가능 → 되도록 홀짝/주차 권장
- **명단 절반** — 기존 회원 명단을 위/아래 반으로

⚠️ 한 사람이 A·B 둘 다 받게 하지 말 것. 한 회원=한 안.

## 무엇을 A/B 해볼까 (바로 쓸 수 있는 아이디어)

### 판매사(구매자) 키우기
| 비교 대상 | A안 | B안 | 볼 숫자 (어디서) |
|---|---|---|---|
| 가입 환영/안내 메시지 | 기본 안내 | 혜택(등급할인) 강조 | **첫 발주율** (어드민 주문) |
| 승인 완료 알림 | "승인됐습니다" | "승인 + 첫 발주 이렇게" 가이드 | 승인→첫 발주 전환 |
| 첫 발주 혜택 안내 | 없음 | "이번 주 무료배송/쿠폰" | 7일 내 발주율 |
| 재발주 리마인드 | 안 보냄 | "재고 떨어질 때쯤" 알림 | **재발주율** |
| 대량주문(엑셀) 안내 | 안 알림 | "엑셀로 한 번에 발주하세요" | 대량주문 사용 수 |
| 알림 보내는 시간대 | 오전 | 오후/저녁 | 알림 후 발주율 |

### 제조사(공급자) 활성화
| 비교 대상 | A안 | B안 | 볼 숫자 |
|---|---|---|---|
| 입점 후 온보딩 | 기본 | "상품 등록 3단계" 가이드 | 상품 등록 수 |
| 출고 독려 메시지 | 안 보냄 | "미출고 N건 있어요" 알림 | **출고 속도(주문→송장)** |
| 정산/인출 안내 | 기본 | "정산 가능 잔액 있어요" | 인출 신청율 |

### 공통/운영
| 비교 대상 | A안 | B안 | 볼 숫자 |
|---|---|---|---|
| 견적/문의 응대 문구 | 짧게 | 단가+대안 제시 | 견적 수락율 |
| 승인 심사 속도 | 평소 | 당일 처리 목표 | 승인→활성 전환 |

## 볼 숫자는 어디서 보나 (어드민 대시보드)

- **가입/승인 수, 리드타임** → `도매 개요` / `셀러 승인`
- **발주 건수·금액, 재발주** → `도매 주문` 관리
- **제조사 출고·정산** → `도매 정산/인출`, 제조사 주문
- **견적/제안 응답** → `견적/제안` 관리
- 숫자는 **실험 시작일·종료일을 메모**하고 같은 기간끼리 비교 (A기간 vs B기간 또는 A그룹 vs B그룹)

---

## 공정 비교 주의 (이러면 결과 왜곡됨)

- ⛔ **같은 기간에 다른 이벤트/프로모션**이 겹치면 → 뭐 때문인지 모름. 한 번에 한 실험.
- ⛔ **명절·연휴·시즌**이 한쪽에만 걸리면 → 주차 번갈아로 보정.
- ⛔ **두 가지 이상 동시에 바꾸기** → 원인 분리 불가. 한 번에 하나만.
- ⛔ **숫자 너무 적을 때**(예: 한 그룹 5명) → 운임. 최소 수십 명 모일 때까지 기간 늘리기.
- ⛔ **중간에 마음에 드는 쪽으로 몰아주기** → 실험 망함. 기간 끝까지.

## 🚫 운영 A/B로 건드리면 안 되는 것 (개발/대표 확인 필요)

이건 운영 재량으로 실험하지 말 것 — 사고/법 위반/손실 위험:

- **공급가(원가)·제조사 이름**을 판매사에게 보여주는 실험 ❌ (영업비밀)
- **결제 금액·수수료·정산 비율·원천징수**를 임의로 바꾸기 ❌ (정산 사고)
- **세금계산서 양식·발급 방식** ❌ (법정 서식)
- **전자계약 서명 차단 우회** ❌
- 가격/MOQ 자체를 바꾸는 실험은 **개발/대표와 먼저 상의** (단순 "안내 문구"는 운영 OK)

→ 운영 A/B는 **메시지·안내·혜택고지·알림 타이밍·응대 문구·노출 순서**처럼 "말과 안내"를 바꾸는 선에서.

## 기록표 템플릿 (복사해서 채우기)

```
[실험 이름]            예) 신규 판매사 환영 메시지 A vs B
- 기간:               ____ ~ ____ (최소 2주)
- 나눈 방법:          예) 가입순번 홀수=A / 짝수=B
- A안(현행):          ____
- B안(바꾼 것):       ____  (한 가지만!)
- 볼 숫자:            예) 가입 후 7일 내 첫 발주율
- 같은 기간 다른 이벤트 있었나?:  없음 / 있음(____)

[결과]
- A안:  대상 ___명, 숫자 ___
- B안:  대상 ___명, 숫자 ___
- 차이:  B가 ___배 (1.3배 이상이면 의미 있음)
- 결정:  A 유지 / B 채택 / 한 번 더 / 보류
- 메모/다음 할 것:
```

### 처음이라면 이걸로 시작
**"신규 판매사 환영 메시지 A(기본) vs B(첫 발주 혜택 강조)"** — 가입순번 홀짝으로 2주, 첫 발주율 비교.
가장 쉽고, 위험 없고, 효과도 눈에 보입니다.
