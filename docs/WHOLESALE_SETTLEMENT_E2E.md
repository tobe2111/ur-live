# 🧾 도매몰 결제 → 정산 staging E2E 검증 체크리스트

> **목적**: 최근 세션에서 정산 계산을 건드린 머니 크리티컬 변경 4건(등급/마진 전환 · 플랫폼 수수료율 ·
> 플러스 연구독 예치금 차감 · 상품별 배송비)이 **단위 테스트로는 통과**했으나, 실제
> `Toss/예치금 결제 → 주문 생성 → 정산 행 → 지급액` 배선은 외부 검증 불가 환경이라 미확인.
> 대표님이 **staging 에서 결제 1회**만 돌려 아래 표와 대조하면 4건이 한 번에 닫힙니다.
>
> 숫자 표는 `src/tests/unit/wholesale-settlement-scenarios.test.ts` 가 실제 코드로 잠가둠 →
> 코드가 바뀌면 테스트가 깨져 이 문서도 같이 갱신됨(드리프트 0).

---

## 0. ⚠️ 먼저 확정 — 플랫폼 수익 모델 (대표님 1줄 확인)

신모델에서 **플랫폼 마진 = 공급가의 N%(기본 10%)** 입니다. 제조사가 나머지 90%를 받습니다.

- 대표님 원안 예시(SSOT): **공급가 8,500 → 플랫폼 850(10%) / 제조사 7,650(90%)** — 현재 코드와 정확히 일치.
- ⚠️ 구모델 대비 변화: 예전엔 플랫폼이 **스프레드 전체(공급가 − 원가)** 를 가졌으나, 지금은 **공급가의 10%만**.
  건강한 상품에선 플랫폼 수익이 줄고 제조사 수령액이 늘어납니다(의도된 방향이면 그대로 진행).

**확인 사항**: 위 "공급가의 10%" 모델이 맞으면 → 진행. "스프레드 전체"를 원하시면 → 알려주세요(정산 1줄 변경).

수수료율은 `/admin/distributor-grades` 에서 조정(`wholesale_platform_commission_pct`, 0~90).

---

## 1. 가격 표시 검증 (결제 전 — 화면만으로 확인)

테스트 상품: **원가 10,000 / 판매가(권장소비자가) 20,000**. 등급별로 로그인해 카탈로그·상세에서 **공급가**가 아래와 같은지 확인.

| 등급 | 보장마진 | 화면 공급가(=청구 단가) | 공식 |
|---|---|---|---|
| 일반 (C) | 15% | **17,000** | max(10000, round(20000×0.85)) |
| 프로 (B) | 30% | **14,000** | max(10000, round(20000×0.70)) |
| 프리미엄 (A) | 38% | **12,400** | max(10000, round(20000×0.62)) |

✅ **체크**: ① 카탈로그 리스트 · ② 상품 상세 · ③ 장바구니 · ④ 주문 미리보기 · ⑤ 최종 청구액 — **5곳 모두 같은 공급가**.
(과거 버그: 일부 화면만 옛값 → 표시가≠청구가. 12곳 배선했으나 실측 1회 필요.)

---

## 2. 결제 → 정산 분배 검증 (결제 1회 후 — DB/지급 대시보드)

위 상품을 **일반 등급으로 1개 주문**(공급가 17,000 청구) 후, 정산 행을 확인.

| 항목 | 기대값 | 공식 |
|---|---|---|
| 유통사 청구(결제) | **17,000** | 공급가 |
| 제조사 정산 | **15,300** | max(원가, round(공급가×90%)) |
| 플랫폼 마진 | **1,700** | 공급가 − 제조사 (= 공급가×10%) |
| 합계 | 17,000 | 누수 0 |

다른 등급 참고: 프로 주문(14,000) → 제조사 12,600 / 플랫폼 1,400 · 프리미엄(12,400) → 제조사 11,160 / 플랫폼 1,240.

✅ **체크**: ① `wholesale_settlements`(또는 정산 대시보드)의 제조사 지급액 = 위 값 · ② 플랫폼 마진 = 공급가−제조사 ·
③ **제조사 정산 ≥ 원가**(원가 하한 — 판매가가 낮아도 제조사가 원가 미만 수령 0).

---

## 3. 원가 하한(degenerate) 안전 검증

판매가가 원가에 가까운 상품(예: **원가 10,000 / 판매가 11,000**)을 프리미엄(38%)으로:

- 공급가 = max(10000, round(11000×0.62)=6820) = **10,000** (원가 하한 바인딩)
- 제조사 = **10,000**, 플랫폼 = **0** ← 손실 차단(플랫폼 0, 제조사 원가 보장). 음수/역마진 절대 없음.

✅ **체크**: 마진이 0이 되는 상품도 결제·정산이 음수 없이 통과.

---

## 4. 환불 역전 검증 (claim-before-credit 대칭)

위 일반 주문을 **환불**(returns) 처리 후:

✅ **체크**: ① 유통사 환불 17,000 — ⚠️ **예치금 잔액(deposit ledger)으로 환원**(`refundDeposit`, Toss 카드취소 아님 — 도매몰은 PG 미사용). 카드 환불이 아니라 **예치금이 17,000 늘었는지** 확인 ·
② 제조사 정산 −15,300 역전(저장된 settlement supply/retail 사용 — pending/available 은 `cancelled`, 이미 지급분은 음수 clawback 행) ·
③ 플랫폼 마진 역전 — `wholesale_orders.margin_total` 이 −1,700 되어(여기선 1,700→0) 줄었는지 직접 확인 ·
④ **재고 복원** · ⑤ 중복 환불 차단(멱등 — 두 번 눌러도 환불·역전 각 1회만).

✅ **부분(라인) 환불 별도 체크**(공급자/제조사 측 라인 선택 환불 — `wholesale-supplier.routes` productIds-scoped): 선택 라인만큼만 ① 예치금 환원 ② 해당 라인 제조사 정산 역전(over-clawback 없음) ③ 주문 status `PARTIAL_REFUNDED`. 전체 환불과 동일하게 멱등.

---

## 5. 플러스 연구독(예치금) 검증 — 별건

⚠️ 도매몰은 PG 미사용 — 구독료는 **예치금(계좌이체 충전 잔액)에서 차감**(Toss 아님).

✅ **체크**: ① 잔액 충분 → 구독 시 `wholesale_plus_annual_fee`(기본 99,000) 차감 + 등급 B + plus_until 갱신 ·
② **더블클릭/동시요청 이중차감 0**(CAS 선점) · ③ 잔액 부족 → 차감·등급변경 없이 충전 유도 · ④ 만료 후 cron 강등(B→C).

---

## 6. 상품별 배송비 검증 — 별건

✅ **체크**: ① 상품별 배송비 입력값이 상세 '배송비' 행에 표시 · ② 묶음(같은 제조사) 주문 시 라인별 유효배송비
**최댓값 1회만** 청구 · ③ 상품별 배송비가 하나도 없으면 = 제조사 정책 배송비(현행과 동일).
