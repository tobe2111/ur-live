## 1. 현재 구현 상태 (사실)

구현됨 (`src/features/supply/api/supply.routes.ts`, mount `/api/supply`):
- 어드민: 상품을 `is_supply_product=1` + `supply_price`(공급가)로 등록 (AdminProductsPage/ProductFormModal)
- 셀러: `GET /products`(카탈로그) → `POST /sample-requests`(신청) → 어드민 승인 → `POST /register`로 **자기 판매가(`seller_price ≥ supply_price`)** 정해 자기 스토어에 복제등록 (`supply_source_id`로 원본 연결, stock 복사)
- 어드민 통계: `SupplySalesTab`/`admin-stats` — `seller_margin = 소매가 − 공급가`, `total_supply_cost = qty × 공급가`
- Cafe24 연동도 `supply_price` import

**핵심 갭**: `supply_source_id`/`supply_price`가 **정산 흐름(auto-settlement/seller-settlements/checkout/payment)에서 미사용**. 통계 추적만. 별도 공급자 엔티티/role 없음(어드민=유일 공급자). 사입/위탁/드랍쉽 구분 없음(stock 복사).

→ 현재 셀러가 공급상품 팔면 **소매가 전액(−플랫폼수수료) 셀러 수령**, 공급가 회수/공급자 지급 정산 **없음**.

## 2. 정산 분배 — 필수 결정 todolist (이걸 정해야 구현 가능)

각 항목은 진짜 갈림길이라 하나라도 미정이면 구현 불가/위험.

- [ ] **D1. 공급자 주체** — 공급자는 누구?
  - (A) 플랫폼(어드민) 자체 = 공급원. 공급가는 원가, 셀러가 소매가로 팔면 차액이 셀러 몫, 공급가는 플랫폼이 회수.
  - (B) 외부 도매상(별도 계정·정산대상). 공급가를 그 도매상에게 지급.
  - (C) 둘 다(플랫폼 직공급 + 외부 도매상 입점).
  - → (B)/(C)면 **공급자 role + 대시보드 + 공급자 정산** 신규 필요.

- [ ] **D2. 자금 흐름 시점** — 판매 시 돈 분배 방식?
  - (A) **즉시 split**: 결제 시 소매가를 공급가→공급자 / 마진→셀러 / 수수료→플랫폼 으로 분할.
  - (B) **후정산(매입)**: 셀러가 소매가 전액 수령 → 주기적으로 공급가를 공급자에게 지급.
  - → 권장 (A) (정산 단순·투명). 기존 ledger/정산 구조와 맞물림.

- [ ] **D3. 플랫폼 수수료 부과 기준**
  - (A) 소매가 전체에 수수료. (B) **셀러 마진(소매가−공급가)에만** 수수료.
  - → 도매몰 관행상 (B)가 공정(셀러가 원가에까지 수수료 안 냄). 현행 5% 기준 재검토.

- [ ] **D4. 재고/이행 모델**
  - (A) **사입**: 셀러가 선구매(재고 보유), 셀러가 배송. (B) **위탁/드랍쉽**: 무재고, 팔리면 공급자가 배송. (C) 공유재고(공급자 재고에서 차감).
  - → 배송 주체 + 재고 차감 위치 결정. 현재 stock 복사는 (A)에 가까우나 결제 없음.

- [ ] **D5. 배송 주체** — (A) 셀러 / (B) 공급자(드랍쉽). 운송장/주소 흐름 결정.

- [ ] **D6. 환불 시 분배 역전** — 환불되면 공급가 회수분도 역전돼야(앞서 고친 누수 패턴과 동일하게 닫기). 부분환불 비례.

- [ ] **D7. 세금 처리** — 공급자 지급 시: 공급자=사업자면 세금계산서(barobill 연동 있음), 개인이면 원천징수. D1=(A)면 내부거래라 단순.

- [ ] **D8. 가격 정책** — `seller_price` 하한=공급가(현행). 상한/MSRP(권장가) 강제? 공급가는 소비자에 숨김(현행 유지).

- [ ] **D9. 정산 주기** — 공급자 지급 주기(즉시/주간/월간). 셀러 정산과 정합.

**최소 결정 셋(이것만 정하면 v1 구현 가능)**: D1, D2, D3, D4, D6.

## 3. 도매몰(Wholesale Mall) 확장 제안 — 단계적

현재 "어드민이 공급상품 올림"을 **다수 공급자 B2B 마켓**으로 확장.

### Phase 1 — 정산 닫기 (갭 해소, 현 기능 완성)
- D1~D6 결정 반영 → 판매 시 공급가 분배(즉시 split) + 환불 역전 + 통계↔실정산 일치.
- 신규 role 없이 어드민=공급자로 먼저 완성(가장 빠름).

### Phase 2 — 공급자(도매상) role 신설
- `suppliers` 엔티티 + 공급자 대시보드(카탈로그 직접 등록·재고·정산 조회).
- 어드민은 공급자 승인/관리. 공급자별 정산(`supplier_settlements`).

### Phase 3 — 소싱 마켓플레이스 UX
- 셀러용 도매 카탈로그 탐색(카테고리·마진율·MOQ·단가구간 필터).
- 발주(PO)/최소주문수량(MOQ)/단가 구간(수량별 할인).
- 공급자↔소매상 메시징·문의.

### Phase 4 — 운영 고도화
- 드랍쉽 자동 배송 연동, 재고 동기화(공유재고), 반품 역물류, 공급자 정산서 PDF.

## 4. 결정 확정 (2026-05-31 사용자)
- **D1 = 외부 도매상 입점** → 공급자 role/계정/대시보드/공급자별 정산 신설 필요 (Phase 2).
- **D2 = 결제 시 즉시 split** → 판매 시 소매가를 공급자/셀러/플랫폼 3분할 (`lib/supply-split.ts` 구현 완료).
- **범위 = 풀 도매몰 (Phase 1~3)**.
- **D3 (미확정 → 권장 기본 채택)**: 마진 기준 수수료 (`feeOnFullRetail=false`). split 헬퍼가 토글 지원 — 추후 변경 1줄.
- **D4 (미정 — 이행/재고)**: ⚠️ **결정 필요**. 권장 기본 = **위탁/드랍쉽(무재고, 공급자 배송)** — 셀러 진입장벽↓, 도매몰 관행. 결정 전 fulfillment(배송·재고차감) 증분 보류.
- D5/D7/D9: D4 확정 후 파생 (배송주체·세금·정산주기).
